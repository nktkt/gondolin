# PyTorch Interop

`NN/Runtime/PyTorch` is Gondolin's bridge layer to and from PyTorch.

PyTorch is used for two practical reasons:

1. **Execution and training** — a practical runtime for trying models, training on
   real data, and checking that shapes and numerics behave as expected.
2. **Moving artifacts** — many workflows already produce PyTorch checkpoints or
   `state_dict`s. The bridge gives those parameters a path back into Lean for
   verification and proofs.

## Export

`Export/` contains reusable exporters and adapters:

- `Export/Core.lean` — shared string utilities and common boilerplate such as imports.
- `Export/IRPyTorch.lean` — the general model code path: exports an `NN.IR.Graph` plus
  a parameter store into a standalone PyTorch module.
- `Export/StateDict.lean` — emits a Python adapter that converts a PyTorch checkpoint
  (`torch.save(model.state_dict(), ...)`) into Gondolin's shape-checkable JSON format.
- `Export/TorchExport.lean` — emits a Python adapter that captures a PyTorch
  `nn.Module` with `torch.export`/FX and writes IR JSON for the supported op subset.

## Import

`Import/` parses JSON-encoded weights and graphs into Gondolin artifacts. Python can
write JSON directly, and Lean can parse it without depending on Python pickle formats.

- `Import/Core.lean` — `parseTensor` turns nested JSON arrays into `Tensor Float s`
  when the JSON shape matches `s`. Small wrappers (`loadWeightsE`, `getTensorE`)
  surface missing-key and shape-mismatch errors.
- `Import/CrownParamstore.lean` — bridges loaded tensors into the graph backend's
  `ParamStore` for node-id-keyed workflows.
- `Import/TorchExport.lean` — parses IR JSON from the generated graph capture adapter
  and accepts only graphs that pass the shared IR validators.

## Examples Live Elsewhere

Architecture-specific loaders and demo round trips live under
`NN/Examples/Interop/PyTorch/{Export,Import}`. Runtime owns the reusable bridge that
examples, verification tools, and downstream projects share.

## What You Can Do Today

- Export PyTorch weights to Gondolin-readable JSON through the generated state-dict
  adapter.
- Parse those JSON tensors in Lean with exact shape checks.
- Capture supported PyTorch `nn.Module` graphs as IR JSON and validate them in Lean.
- Load supported verification model families such as PINNs/FNOs through the example
  interop loaders.
- Emit readable PyTorch code from a Gondolin `NN.IR.Graph` and `ParamStore`.

What is deliberately **not** claimed: Lean does not prove PyTorch or CUDA kernels
correct, and it does not parse `.pt`/`.pth` pickle/zip checkpoints directly.
PyTorch is the external loader for those files; Gondolin checks the JSON artifact
it receives.
