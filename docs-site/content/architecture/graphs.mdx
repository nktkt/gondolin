# Dependency Maps

These pages explore how TorchLean is organized: which modules import each other,
where the main hubs are, and how the API, runtime, proof, and verification layers
connect.

A module edge means one Lean file imports another. It does **not** mean every theorem
in the source file depends on every theorem in the target file. Treat the maps as a
structural orientation tool, not a measure of proof dependency.

## Repository Shape

A repository-shape view summarizes:

- total modules and import edges,
- public vs. private imports,
- the longest chain through direct imports (a rough measure of layering depth).

These counts are generated from the Lean source tree during the site build and are
useful scale indicators only — not semantic proof-dependency measurements.

## Codebase Snapshot

The snapshot reports Lean source lines, code lines, comment/blank lines, declaration
headers, and theorem/lemma headers. It also breaks down source lines by top-level
layer (`NN.Spec`, `NN.Runtime`, `NN.MLTheory`, `NN.Floats`, etc.).

## Module Explorer

The explorer lets you:

- search modules by substring,
- filter by source layer or destination layer,
- click any module to see its direct imports and direct importers,
- see whether each edge is public or private and which `.lean` file declares it.

Click through hubs in the import-graph: the most-imported modules act as foundation
nodes, and the most-importing modules act as umbrella files.

## Layer Flow

The layer-flow view aggregates edges by top-level layer (`NN.Spec` → `NN.Spec`,
`NN.Runtime` → `NN.Spec`, etc.) and ranks them by edge count. It is the fastest way
to see whether two subsystems talk to each other at all, and roughly how heavily.

## Import Hubs

Two leaderboards summarize:

- **Most Imported** — fan-in for foundation modules.
- **Most Importing** — fan-out for umbrella and entrypoint files.

For the raw numbers, see the [Dependency Audit](./dependency-audit).
